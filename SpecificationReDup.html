Overview

Goals

  1. Explore GitHub integration and API
  2. Practice a full-cycle application development process

Out of Scope

  1. UI/Front-end: the main value Application provides is with it’s functionality and UX part, namely accessing user’s GitHub repositories and copying it’s code to user’s profile.
  2. Authentication/Authorization: we’ll use SnapEngage for now, and review if the test works.
  3. Hosting application on a specific domain.

People and Roles

  1. GitHub User: GitHub user who is interested in getting a copy of this solution’s repository on his GitHub account.
  1. Application owner: A person who owns an application and wants to distribute it among GitHub users.

Context

Use Cases

  1. Users want to:
  1. Copy this solution’s repository without efforts at all
  2. Do not share their authentication credentials(password) with third party
  1. Application owner wants to:
  1. Avoid sharing secrets with application users
  2. Spend less time on manual app setup

Assumptions

  1. UX is more valuable than UI for application users
  2. Users would prefer having a complete copy of app repo, rather than fork
  3. Create repository from template GitHub feature would not be discontinued, but rather go live
  4. App owner would not mind manually creating GitHub app and initial terraform setup along with providing some secrets as environment variables

Proposal

It is suggested to:

  • Use GitHub OAuth app to request access to user public repositories
  • Use GitHub preview feature of creating repository from template.
  • Prioritise UX over UI for users by providing single button solution
  • Host application in AWS to reduce cost and effort for infrastructure setup and maintenance
  • Allow setup of AWS resources using terraform
  • Leverage usage of CICD provider CircleCI for ease of development
  • Have an SLA of 99.99%
  • Keep in mind some Improvements that can be made

GitHub application

As a convenient way GitHub OAuth application provides a seamless way of authorizing users without requiring them to share their credentials, while giving them trusted GitHub interface to validate the application.

Create repository from template

Though it is a preview feature, GitHub feels very confident about this and plans to develop it further. A concern regarding this feature discontinuation is valid, and additional monitoring of the matter would be beneficial. As a fallback mechanism it is possible to use Fork functionality, which provides similar user experience.

User Experience

As Users are more interested in a working solution that provides them value of seamless copying app repository, they would be less  interested in flashy UI with bells and whistles and more interested in an easy workflow that get things done. That is why providing single page with simple text and a button that does the trick is a preferable solution for starters.

AWS infrastructure

It makes sense to leverage the usage of AWS infrastructure as the cost and development time would be considerably lower, compared to hosting on dedicated servers. Also AWS provides a set of services that allows us to implement critical parts of application fast and have a great space for fast expansion after initial version. They include:

  • AWS Lambda
  • API Gateway
  • Cloudwatch(logging)
  • AWeSome terraform provider for setting up infrastructure as a code

This would allow to have the following benefits

  • Access to latest version of Python
  • Ability to use other backend languages for writing business logic
  • Scalability
  • Availability
  • Accessibility
  • Usage of other AWS services, should we decide to invest more in the solution

Terraform

Infrastructure as a code allows to store the state of infrastructure and eliminate manual effort for setting it up. Saves a lot of time and effort, allows have a description of used resources in a comprehensive form. Terraform is one of the most popular infrastructure as a code providers. That gives access to good integration support, a lot of comprehensive guides and technical articles, as well as  community support for fixing issues and adding new features. That also allows to implement new features of services that are being setup faster, allowing to use them shortly after being released.

CICD

CICD allows to have releases more frequent and in a much more controllable way, along with improving developer experience by testing and deploying changes shortly after they have been committed. CircleCI is a great tool, which allows seamless configuration, access to “orbs”, that allow easier building process for repetitive actions and also a free tier of builds that can be used on early stages of development.

SLA

It is expected that SLA of application would be no less, then the

AWS General Service Commitment  SLA: 99.99%
